# Purpose

Use this skill for tables and dense data interfaces.

This skill is for:
- product tables
- admin tables
- customer record lists
- operational data grids
- searchable and filterable lists
- row actions
- bulk selection flows
- dense but readable data surfaces

# Use this skill when

Use this skill when the request involves:
- building a table
- improving filtering or sorting UX
- cleaning up dense data layouts
- making tables more readable and operational
- improving pagination, actions, or scanability
- refining row detail and list interactions

# Do not use this skill when

Do not use this skill for:
- chart-only dashboards
- generic page design
- tiny visual tweaks with no table significance
- pure backend pagination implementation with no UX surface

# Table philosophy

A good table is not just rows and columns.
It is a decision and action surface.

It must support:
- quick scanning
- clear statuses
- obvious actions
- understandable sorting/filtering
- low-friction navigation
- graceful empty/loading/error states

# Workflow

## 1. Define the table purpose

Determine:
- what the user is looking for
- what they compare most often
- what actions they take from the table
- what columns are most important
- what filters/search matter
- whether row-level or bulk actions exist

## 2. Improve data readability

Prefer:
- strong column prioritization
- readable spacing
- aligned numeric/status fields
- meaningful truncation strategy
- status emphasis where relevant
- no overcrowding without purpose

## 3. Improve controls

Review:
- filtering
- sorting
- pagination
- search
- row actions
- selection/bulk actions
- detail navigation

Ensure these controls feel integrated and understandable.

## 4. Handle states

Include:
- loading
- empty
- no-results after filters
- error
- partial data if relevant
- disabled bulk actions if relevant

## 5. Review responsiveness and degradation

Check:
- what happens on narrower widths
- whether columns collapse gracefully
- whether row details need alternate presentation
- whether actions remain usable

## 6. Final summary

Summarize:
- table purpose
- columns/actions/controls improved
- scanability/readability gains
- state handling added
- remaining operational gaps if any

# Quality bar

The table should feel:
- fast to scan
- operational
- premium
- clear under density
- trustworthy for action-heavy usage
